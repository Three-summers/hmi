## 按钮设计限制要求
### 1. 尺寸要求
- **触摸屏使用场景**：
  - 按钮尺寸必须足够大，以支持可靠触控操作。
  - **最小尺寸**：按钮在最短边上的长度**不应小于 1.5 厘米**。
  - 若因布局限制无法满足该尺寸，则**必须增加按钮之间的间距**，以防止误触。

- **键鼠使用场景**：
  - 导航面板与命令面板中的按钮尺寸可适当减小。
  - **推荐尺寸范围**：约 **1.0 至 1.25 厘米**（最短边）。
  - 信息面板中的按钮尺寸应**与面板整体比例协调**，并相应放大以保持视觉一致性。

---

### 2. 行为类型
按钮应支持以下两种交互行为类型：
#### 2.1 瞬时类型
- 用户点击时，按钮**短暂进入“按下”状态**，随即自动返回“弹起”（未选中）状态。
- 适用于触发一次性操作（如“提交”、“刷新”）。

#### 2.2 双态类型
- 用户点击后，按钮**保持“按下”状态**，直至满足以下任一退出条件：
  - 用户**再次点击该按钮**；
  - 用户**点击同一组内的另一个按钮**（适用于单选按钮组）；
  - 用户**点击界面其他区域**；
  - **软件逻辑主动控制**退出（如任务完成、状态变更）。
- **视觉反馈要求**：
  - 按下状态必须通过**阴影、交叉阴影、纹理变化**等方式明确标识；
  - 视觉效果**不得遮挡按钮标签文本**。

---

### 3. 文本规范
- 所有按钮标签文本应采用 **“首字母大写”格式**（Title Case）。
  - 示例：`Start Process`、`Save Configuration`、`Cancel Operation`
- **禁止使用全大写**（如 `SAVE` 或 `CANCEL`），以提升可读性与专业性。

---

### 4. 突出显示
- **目的**：用于吸引用户注意或指示状态。
- **实现方式**：
  - 可采用**颜色、纹理**，或**两者结合**；
  - 突出显示区域应**环绕按钮或显示对象边缘**，形成视觉边界。
- **限制要求**：
  - 突出显示效果**不得遮挡或隐藏**其包围的按钮或显示对象内容。
- **颜色语义规范**：

| 状态/用途       | 推荐颜色 |
| ----------- | ---- |
| 报警（Alarm）   | 鲜红色  |
| 警告（Warning） | 鲜黄色  |
| 正在处理 / 未完成  | 中蓝色  |
| 需要或请求用户注意   | 中绿色  |
> 注：颜色应具有足够对比度，确保在各种光照和显示条件下均可清晰识别。

## 对话框设计规范
### 1. 用途
对话框是**临时辅助窗口**，用于：
- 显示补充或详细信息（如系统监控数据）；
- 向用户索取输入（如填写参数）；
- 报告错误或状态（如“连接失败”）。

> ✅ 示例：主界面只显示“温度：60°C”，点击后弹出对话框显示“传感器 ID、历史曲线、报警阈值”等细节。

---

### 2. 显示与交互规则
#### 2.1 触发与覆盖
- 仅由**用户操作触发**（如点击按钮或对象）；
- 弹出时**覆盖部分信息面板**，但**不得遮挡标题面板**（顶部区域）；
- 若由**信息面板或命令面板**中的对象触发，则这两个面板**全部禁用**，直到对话框关闭；
- **标题面板和导航面板始终保持可用**（用户可查看当前位置或切换页面）；
- 关闭后，**信息面板内容自动刷新**。

> ⚠️ 禁止：对话框弹出后仍允许用户操作触发它的面板，可能导致状态冲突。

#### 2.2 不可移动与缩放
- 对话框**固定位置和尺寸**，用户**不能拖动或调整大小**；
- 允许在标题栏提供“×”关闭按钮，其行为等同于 **Cancel**。

---

### 3. 结构与布局
对话框必须包含三部分：
1. **标题栏**：文字说明来源或目的（如“确认删除？”、“网络设置”）；
2. **内容区**：自由布局，用于显示文本、输入框、列表等；
3. **底部按钮区**：水平排列操作按钮。

#### 按钮布局规则
- **主要操作按钮**（如 OK、Cancel、Yes、No、Close）**居中对齐**；
- **辅助功能按钮**（如 Apply、Log-out）**靠右对齐**，并与主按钮**视觉分离**（如留空或加竖线）；
- 若系统限制无法实现，可调整对齐方式，但**所有对话框必须保持一致**。

---

### 4. 按钮行为定义

| 按钮类型 | 使用场景 | 行为 | 特殊规则 |
|--------|--------|------|--------|
| **OK** | 有输入或选择项 | 接受当前值（含默认值）并关闭 | 若有必填项未完成，**必须禁用** |
| **Cancel** | 可撤销操作 | 关闭对话框，**恢复原状态** | 适用于可回退的场景 |
| **Yes / No** | 对话框内容为问题（如“是否保存？”） | Yes：确认；No：拒绝并关闭 | 必须成对出现 |
| **Close** | 仅展示信息（无操作）<br>或**无法恢复原状态** | 关闭，**不执行任何操作** | ❌ 不得用 OK 代替<br>✅ 示例：“系统已重启”提示 |
| **Apply** | 设置类对话框（如音量、颜色） | 应用当前设置，**不关闭对话框** | 初始禁用；点击后再次禁用，直到有新修改 |
> 💡 提示：
> - “无法恢复原状态”指操作已部分执行（如已发送指令），此时用 **Close** 而非 Cancel。
> - Apply + OK 组合常见于高级设置：Apply 试用，OK 确认并退出。

### 5. 类型
对话框的整体布局统一为三部分：
- **顶部**：对话框标题（Dialog Title）
- **中部**：显示区域（Display Area）
- **底部**：对话框窗口控制按钮（Dialog Window Control Buttons）

根据用途和内容，对话框分为以下三种类型：

---

#### 5.1 信息对话框
- **用途**：仅用于**单向展示信息**，不涉及用户输入或决策。
- **显示区域**：仅包含**消息文本**（Message Area）。
- **按钮要求**：
  - 必须使用 **`Close`** 按钮；
  - **禁止使用 `OK` 按钮**（即使内容看似“确认”，也不应使用 OK）。
- **示例**：
  - “系统已成功启动。”
  - “当前版本：v2.1.0”

> ✅ 正确：`[Close]`
> ❌ 错误：`[OK]`

---

#### 5.2 数据输入/选择对话框
- **用途**：请求用户**输入数据**（如文本、数值）或**从选项中选择**（如列表、复选框）。
- **显示区域**：包含输入控件（文本框、下拉菜单、单选按钮等）。
- **特殊要求**：
  - 如果系统**无物理键盘或数字键盘**，且用户需输入字符，则**必须在对话框内提供屏幕键盘的可视化表示**（如虚拟键盘）。
- **按钮要求**：
  - 必须包含 **`Cancel`（取消）** 和 **`OK`（确定）** 按钮；
  - `OK` 按钮在必填项未完成时应**禁用**。
- **实现条件**：
  - 仅当存在需要用户输入或选择的显示对象时，才应启用此类型对话框。

> ✅ 示例：
> - “请输入设备名称：[\_\_\_\_\_\_\_\_\_\_]” → `[Cancel] [OK]`
> - 无键盘设备上弹出带虚拟键盘的输入框。

---

#### 5.3 消息对话框
- **用途**：
  - 向用户显示**通知、警告、错误**等状态信息；
  - 或**请求用户确认操作**（如“确定要删除文件吗？”）。
- **显示区域布局**：
  - **左侧**：一个图标；
  - **右侧**：消息文本（紧邻图标右侧）。
- **图标规范**（推荐使用操作系统标准图标）：

| 类型              | 图标     | 说明                  |
| --------------- | ------ | ------------------- |
| **Information** | ℹ️     | 普通通知（如“保存成功”）       |
| **Progress**    | 🔄 / ⏳ | 后台正在进行操作（如“正在同步…”）  |
| **Attention**   | ⚠️     | 需要用户注意或确认（可为陈述句或问句） |
| **Error**       | ❌ / 🛑 | 严重错误或操作失败（如“权限不足”）  |
- **禁止使用问号（❓）图标**，即使消息是疑问句（如“是否退出？”），也应使用 **Attention（⚠️）图标**。
- **按钮规则**（根据消息类型）：
  - **Information / Progress**：使用 **`Close`** 按钮；
  - **Attention / Error**：
    - 若为**陈述句**（如“无法删除只读文件”）→ 使用 **`OK`** 或 **`OK / Cancel`**；
    - 若为**疑问句**（如“是否覆盖文件？”）→ 使用 **`Yes / No`**，可选加 **`Cancel`**。
- **实现前提**：仅当系统**具备提供该消息所需信息的能力**时，才应显示此类对话框。

> ✅ 示例：
> - “是否保存更改？” + ⚠️ → `[Yes] [No] [Cancel]`
> - “网络连接中断” + ❌ → `[OK]`
> - “正在导出数据…” + 🔄 → `[Close]`

---

#### 5.4 总结对比表

| 类型 | 是否需用户操作 | 图标 | 按钮 | 典型场景 |
|------|----------------|------|------|--------|
| **信息对话框** | 否 | 无 | `Close` | 状态提示、版本信息 |
| **数据输入/选择对话框** | 是 | 无 | `Cancel` / `OK` | 填写参数、选择选项 |
| **消息对话框** | 视情况 | ℹ️ / 🔄 / ⚠️ / ❌ | `Close`、`OK/Cancel`、`Yes/No` | 错误提示、操作确认 |

---

### 6. 关键原则总结
- **临时性**：对话框不是主界面，用完即关；
- **模态性**：阻止用户操作触发源，防止状态混乱；
- **一致性**：按钮语义、布局、行为全系统统一；
- **安全性**：Cancel/No 必须能安全回退，OK/Yes 必须明确用户意图。

### 1. 基础网络导航模型（强制性）
#### 1.1 设计目标
- 最小化用户操作步骤和完成任务所需时间；
- 避免传统树形菜单中“逐级返回”的繁琐导航。

#### 1.2 结构特点
- 所有功能区域（如 Jobs、Alarms、Recipes、Setup 等）处于**同一层级**；
- 用户可**随时横向切换**任意功能区域，无需返回上级菜单。

#### 1.3 视图与显示规则
- **每个功能区域仅包含一个视图**（例如：“Alarms”只显示当前报警列表）；
- 如需查看详细信息（如某条报警详情），应通过**对话框**展开，**不得新增视图**；
- **整个界面一次仅显示一个信息面板**：
  - 用户选择某个功能区域后，仅显示其对应的信息面板，其他区域内容自动隐藏。

> ✅ 示例：
> 点击 “Recipes” → 显示配方列表；
> 点击 “Alarms” → 配方面板消失，显示报警列表。

---

### 2. 带子导航的网络导航模型（条件性）
> ⚠️ **启用条件**：仅当**至少一个功能区域需要多个视图**时，才可使用此模型。

#### 2.1 一致性要求
- 一旦任一功能区域使用多视图，**所有功能区域必须统一采用同一种子导航方式**，二选一：
  1. **标签页（Tabs）**：单行横向标签，位于信息面板顶部（如 `[实时] [历史] [日志]`）；
  2. **专用子导航按钮区**：在屏幕独立区域（如信息面板上方或侧边）设置一组仅用于视图切换的按钮。

> ❌ 禁止：部分功能区用标签，另一部分用按钮。

#### 2.2 功能组织原则
- 每个功能区域应围绕**单一用户任务**组织多个视图；
- 视图内容应逻辑相关（如“监控”区域包含：实时数据、趋势图、诊断信息）；
- 目标是**减少跨功能区域跳转**，提升操作效率。

#### 2.3 显示规则
- **整个界面始终只显示一个信息面板**；
- 在该面板内，**始终只显示一个子视图**（用户切换标签或按钮时，其他视图隐藏）；
- 视图内的详细信息仍通过**对话框**展示。

> ✅ 示例：
> 功能区域：“Monitoring”
> 子视图：`[Overview] [Trends] [Events]`（通过标签切换）
> 点击趋势图中某点 → 弹出对话框显示具体数值和时间。

---

### 3. 通用设计原则（适用于两种模型）

| 原则         | 说明                |
| ---------- | ----------------- |
| **扁平化导航**  | 所有功能区域同级，支持任意横向跳转 |
| **单面板显示**  | 任何时候只显示一个信息面板     |
| **详情用对话框** | 不通过新页面或嵌套视图展示细节   |
| **任务导向分组** | 功能区域按用户实际工作流程组织   |
| **避免过度复杂** | 若无需多视图，应坚持使用基础模型  |

## 显示布局
### 1. 显示布局设计目标
- 布局专为**触摸屏操作优化**，**无需键盘或鼠标等指向设备**；
- 屏幕被划分为若干**矩形面板**，每个面板对应用户在以下任务中的信息需求：
  - 监控与控制工艺过程；
  - 设备维护与维修；
  - 其他相关操作任务。

> ✅ 目的：让用户在触摸操作时，信息与控件按任务逻辑分组，减少误触和认知负担。

---

### 2. 基础布局（强制性）
#### 2.1 面板组成与位置
- 界面**必须包含四个面板**，按图示方式排列（见原 Figure 8）：
  1. **标题面板（Title Panel）**：顶部，显示系统名称、当前页面或状态；
  2. **信息面板（Information Panel）**：中央主区域，显示数据、状态、图表等；
  3. **命令面板（Command Panel）**：**默认位于右侧**，包含操作按钮（如启动、停止、确认）；
  4. **导航面板（Navigation Panel）**：通常位于左侧，用于切换功能区域（如 Jobs、Alarms、Setup）。

> ⚠️ **强制要求**：除非采用增强布局（见第 3 节），否则命令面板**必须在右侧**。

#### 2.2 布局规则
- 所有面板**紧密平铺（edge-to-edge）**，无间隙；
- **仅规定相对位置**（如“命令面板在右”），不强制尺寸或边框；
- 面板**可有或没有可见边框**，由设计决定。

#### 2.3 主显示窗口的安全限制
- 若该界面是**主显示界面**（通常位于设备正面，用于日常操作）：
  - **禁止显示操作系统窗口边框**（如标题栏、关闭×、最小化按钮）；
  - **禁止允许用户关闭、最小化或随意移动窗口**；
  - **原因**：防止用户误操作导致界面“消失”，引发安全隐患（如无法监控关键参数）。
- **例外**：具有高级权限的已登录用户（如工程师）**可允许调整窗口大小**，但**仍不得关闭或最小化**。
- **次要实例**（如远程维护终端、调试节点上的副本）**可启用完整窗口框架**（含关闭、缩放等）。

> ✅ 示例：
> - 操作员在设备前使用的主屏 → 无“×”关闭按钮；
> - 工程师在办公室远程连接 → 可缩放窗口，但不能关闭主进程。

---

### 3. 增强布局（推荐）
#### 3.1 支持左撇子用户
- **强烈建议**提供选项，允许用户将**命令面板从右侧切换到左侧**（见原 Figure 9 (b)）；
- **目的**：左撇子用户用左手操作右侧命令面板时，手臂会遮挡屏幕中央信息；移到左侧可避免此问题。

#### 3.2 布局变体
- **(a) 默认布局**：命令面板在右（适用于右利手）；
- **(b) 左手优化布局**：命令面板在左，其余面板相应调整位置。

> 💡 实现建议：
> - 在用户设置中提供“命令面板位置”选项（左/右）；
> - 切换后，导航面板自动移到对侧，保持信息面板居中。

### 4. 总结对比

| 项目          | 基础布局         | 增强布局          |
| ----------- | ------------ | ------------- |
| **是否强制**    | ✅ 是          | ❗ 推荐（非强制）     |
| **命令面板位置**  | 固定右侧（除非启用增强） | 可选左侧或右侧       |
| **适用场景**    | 所有系统         | 特别适合多用户或左利手环境 |
| **主界面窗口控制** | 禁用关闭/最小化     | 同基础布局（安全限制不变） |

## 标题面板
### 1. 标题面板概述
- **位置**：位于界面顶部，横跨整个宽度，**始终显示**；
- **作用**：提供用户**始终需要可见的关键状态与操作入口**；
- **基本内容**（根据系统能力动态包含）：
  - 当前视图名称
  - 日期/时间
  - 消息显示区
  - 主机通信状态（如支持）
  - 登录/登出按钮（如启用安全功能）
- **可选内容**（推荐）：
  - 企业标识（Logo）
  - 关键参数（如温度、压力）
  - 声光报警静音按钮
  - 设备方向图示
  - 虚拟“灯塔”状态指示（如红/黄/绿）等

> ✅ 设计原则：只放**必须常显**的信息，避免干扰主操作区。

---

### 2. 基础信息（强制性）
所有系统**必须包含以下三项**，且**布局位置固定**（见原 Figure 10）：

| 元素 | 位置 | 说明 |
|------|------|------|
| **日期/时间显示** | 标题面板**左上角** | 实时更新 |
| **当前视图名称** | 紧邻日期/时间**右侧** | 如 “Alarms”、“Recipe Setup” |
| **消息显示区** | 位于上述两项**正下方**（标题面板下半部分） | 用于显示系统通知、警告等简短消息 |
> ⚠️ **强制要求**：这三者的相对位置**不得更改**。

---

### 3. 条件性信息（按需启用）
#### 3.1 主机通信状态（条件性）
- **启用条件**：设备支持与上位机（如 MES、SCADA）通信；
- **必须显示在标题面板最左侧**（位于日期/时间之前）；
- **可包含内容**：
  - 通信是否激活（Active/Inactive）
  - 连接状态（Connected / Disconnected）
  - 控制模式（Local / Remote）
- **具体内容**需遵循所用通信协议的要求（如 OPC UA、Modbus 等可能规定特定状态字）。

> ✅ 示例：`[HOST: REMOTE • CONNECTED]`

#### 3.2 登录/登出按钮（条件性）
- **启用条件**：系统启用了用户权限或安全功能；
- **位置**：标题面板**右上角**（强制）；
- **行为**：
  - **未登录时**：按钮显示为 **“Login Here”**；
  - **已登录时**：显示**当前用户名**（如 “Operator_A”）；
- **点击后**：
  - 弹出登录/登出对话框；
  - 若系统要求高安全性，**可临时禁用整个界面**（包括导航面板），直到操作完成。

> 🔒 安全建议：在登录对话框弹出时禁用其他操作，防止权限绕过。

---

### 4. 推荐的附加信息
在基础+条件内容之上，**推荐**在标题面板中加入以下元素（见原 Figure 12）：

| 元素              | 说明                                |
| --------------- | --------------------------------- |
| **企业标识 / Logo** | 品牌识别，通常靠左或居中                      |
| **虚拟灯塔显示**      | 用颜色图标表示设备整体状态（绿=正常，黄=警告，红=停机）     |
| **关键参数**        | 如“压力：5.2 bar”、“温度：85°C”，仅限最关键 1–2 项 |
| **报警静音按钮**      | 快速关闭声音报警（不影响视觉报警）                 |
| **设备方向图示**      | 在移动或可旋转设备中指示当前朝向                  |
> 💡 布局建议：附加信息应**不遮挡强制内容**，通常放在右侧或中部空余区域。

---

### 5. 报警按钮（特殊说明，不推荐用于新设计）
- **旧设计可能包含**一个 **“Alarms” 按钮**直接放在标题面板；
- **若使用此按钮**，则：
  - **导航面板中不得再出现“Alarms”导航项**（避免重复）；
  - 点击后**必须通过对话框显示报警**，**不能打开新的信息面板+命令面板组合**；
- **新系统应避免此设计**，推荐将报警作为独立功能区域放在导航面板中。

> 🚫 新项目建议：**不要在标题面板放 Alarms 按钮**。

---

### 6. 总结：标题面板内容优先级

| 类型 | 是否必须 | 位置要求 | 说明 |
|------|--------|--------|------|
| 日期/时间、当前视图名、消息区 | ✅ 强制 | 固定（左上→右，消息在下） | 所有系统 |
| 主机通信状态 | ⚠️ 条件性 | 最左侧 | 仅当支持通信 |
| 登录/登出按钮 | ⚠️ 条件性 | 右上角 | 仅当启用安全 |
| Logo、灯塔、关键参数等 | 💡 推荐 | 灵活（不干扰强制项） | 提升可用性 |
| 标题面板 Alarms 按钮 | ❌ 不推荐 | — | 仅限旧系统兼容 |

## 导航面板
### 1. 导航面板概述（强制性）
- **位置**：位于屏幕**底部**，横跨整个宽度；
- **始终可见**：即使在对话框弹出时也不得被遮挡；
- **组成**：由一组**导航按钮**构成，每个按钮代表一个功能区域（如 Jobs、Alarms、Recipes 等）；
- **按钮格式**：
  - **必须包含文本标签**；
  - **可选包含图标**；
  - 若**无图标**，标签**居中显示在按钮上**；
  - 若**有图标**，标签**居中显示在图标下方**。

> ✅ 目的：确保用户无论当前操作状态如何，都能快速、可靠地切换功能区域。

---

### 2. 必需的导航功能（强制性）
#### 2.1 报警与安全访问必须始终可用
- 用户**必须能立即响应报警和警告**，即使当前有对话框打开；
- **导航面板不得被任何对话框遮挡**；
- 用户**无需关闭或操作当前对话框**，即可点击导航按钮（如“Alarms”）；
- 导航后返回原视图时，**原对话框必须恢复原状**（内容、状态不变）。

#### 2.2 特殊例外
- **登录/登出对话框**可为**模态对话框**（modal），此时可临时禁用导航面板；
- 此为**唯一允许的例外**。

> ⚠️ 安全原则：关键操作（如处理报警）不能因界面状态被阻断。

---

### 3. 条件性导航功能（推荐行为）
- 除登录/登出场景外，**导航面板应始终可点击**；
- 用户应能**从任何界面状态直接跳转到任意功能区域**；
- 返回时，系统应**恢复之前的视图和对话框状态**。

> 💡 实现建议：采用非模态对话框，或在模态对话框下仍允许导航（通过覆盖式对话框机制）。

---

### 4. 导航面板布局（强制性）
#### 4.1 按钮数量与占位
- **最多建议 10 个按钮**；
- 预留 **3 个 “(Future)” 占位符**（见原 Figure 13），用于未来扩展；
- 新增按钮应插入在 **“Setup” 和 “Alarms” 之间**。

#### 4.2 按钮顺序规则
- 默认按**使用频率从高到低**从左到右排列；
- **最常用功能在最左侧**，最少用在右侧。

> 示例顺序（默认）：
> `Jobs → System → Recipes → Setup → (Future) → (Future) → (Future) → Alarms → Help`

---

### 5. 报警与帮助按钮的特殊位置（条件性）

- **Alarms 按钮**：**倒数第二位**（next to right-most）；
- **Help 按钮**：**最右侧**（right-most）；
- **目的**：确保这两个关键按钮位置**永不改变**，即使新增其他功能；
- **Alarms 按钮周围应留更大间距**，便于快速、准确点击（尤其在紧急情况下）。

> ✅ 示例：
> `... → Setup → (Future) → Alarms [宽间距] Help`

---

### 6. 导航按钮标签（条件性）
#### 6.1 标签要求
- 每个功能区域对应一个导航按钮；
- **文本标签为强制项**，图标为推荐项；
- 标签应清晰反映功能内容。


#### 6.2 推荐功能区域与标签（见原 Table 1）

| 功能区域        | 推荐主标签   | 可选替代标签                                 |
| ----------- | ------- | -------------------------------------- |
| **Jobs**    | Jobs    | Lot, Operations, Run, Processing, Main |
| **System**  | System  | Overview, Status, Service, Maintenance |
| **Recipes** | Recipes | （无推荐替代）                                |
| **Datalog** | Datalog | History, Logs, Data, Analysis          |
| **Setup**   | Setup   | Configuration, Options                 |
| **Alarms**  | Alarms  | （固定位置，见 5）                             |
| **Help**    | Help    | （固定位置，见 5）                             |
> 📌 注：表格从上到下对应按钮从左到右顺序。

#### 6.3 旧版布局（不推荐用于新设计）
- 允许但**不推荐**的顺序：
  `System → Jobs → Equipment Setup → Recipes → History → Maintenance → Configuration`
- 若使用，仍需将 **Alarms 和 Help 放在最右两位**。

---

### 7. 导航按钮状态指示（Saliences，条件性）
按钮通过**颜色高亮**（salience）传达状态：

#### 7.1 三种状态颜色
| 状态                       | 颜色                  | 说明                 |
| ------------------------ | ------------------- | ------------------ |
| **当前所在功能区**              | 中蓝色（Medium Blue）    | 按钮呈“按下”状态 + 蓝色高亮   |
| **其他功能区有未完成任务**（如打开的对话框） | 中蓝色（Medium Blue）    | 按钮“未按下”但高亮，提醒用户    |
| **新警告（Cautions）**        | 饱和黄色（Bright Yellow） | 仅显示在 **Alarms 按钮** |
| **新报警（Alarms）**          | 饱和红色（Bright Red）    | 仅显示在 **Alarms 按钮** |

#### 7.2 优先级规则
- 若同时存在警告和报警，**只显示红色**（报警优先级更高）；
- 一旦用户确认/清除所有报警和警告，Alarms 按钮恢复为中蓝色（如有未完成任务）或无高亮。

#### 7.3 Jobs 按钮特殊状态
- 可显示**中绿色高亮**，表示设备处于关键状态，如：
  - “Ready to Load”
  - “Ready to Unload”
  - “Ready to Run”
- 若同时有未完成任务（蓝色）和就绪状态（绿色），**蓝色优先显示**（任务状态更重要）。

> ✅ 示例场景：
> 用户在 “Recipes” 视图，但 “Jobs” 有打开的对话框 → “Jobs” 按钮显示中蓝色（未按下）；
> 同时设备“Ready to Run” → 仍显示蓝色，绿色不覆盖。

---

### 8. 子导航（条件性）
当某个功能区域包含**多个视图**时，必须使用以下**两种子导航方式之一**，且**全系统保持一致**。

#### 8.1 方式 A：标签页（Tabs）
- 在信息面板顶部显示**单行横向标签**；
- 点击标签切换视图，对应的信息面板和命令面板随之更新；
- **即使某功能区只有一个视图，也应显示一个标签**（保持一致性）；
- 支持命令面板在左或右（见原 Figure 17）。

#### 8.2 方式 B：独立子导航面板（Buttons）
- 在信息面板上方或下方设置**专用子导航按钮区**；
- 按钮用于切换视图；
- **位置和尺寸必须在所有功能区保持一致**；
- 即使某功能区只有一个视图，也应保留该面板（可为空）。

#### 8.3 设计原则
- **子导航必须与命令面板分离**；
- **目的**：
  1. 避免用户混淆“视图切换”与“操作命令”；
  2. 保持所有视图的**信息面板宽高比一致**，防止内容“跳动”；
  3. 减少命令面板按钮数量，避免多列布局。

> ✅ 最佳实践：**优先使用标签页（Tabs）**，界面简洁、用户熟悉。

---

### 9. 总：导航面板关键规则速查

| 项目 | 要求类型 | 说明 |
|------|--------|------|
| 面板位置 | 强制 | 屏幕底部，始终可见 |
| 按钮标签 | 强制 | 必须有文本，图标可选 |
| Alarms/Help 位置 | 条件强制 | 倒数第二、最右，间距加大 |
| 按钮顺序 | 强制 | 按使用频率，Alarms/Help 固定 |
| 报警访问 | 强制 | 对话框不阻断导航 |
| 状态高亮 | 推荐 | 蓝（任务）、黄（警告）、红（报警）、绿（就绪） |
| 子导航方式 | 条件 | 全系统统一用 Tabs 或按钮面板 |
| 最大按钮数 | 推荐 | ≤10 个 |

## 命令面板
### 1. 命令面板概述（强制性）
- **位置**：
  - **默认位于信息面板右侧**；
  - 若系统支持**左手操作模式**（见增强布局），可切换至**左侧**；
- **方向**：**垂直排列**的按钮列；
- **作用**：提供与**当前视图直接相关**的**通用或全局命令**。

> ✅ 设计目标：让操作员一眼识别哪些命令适用于当前任务，避免误操作。

---

### 2. 命令内容规则（强制性）
#### 2.1 仅包含视图相关命令
- 命令面板中的按钮**必须仅适用于当前信息面板所显示的视图**；
- 不同视图应有**各自对应的命令面板**；
- 若某视图**无通用命令**，则命令面板**应为空**（不显示按钮）。

> ✅ 示例：
> - “报警列表”视图 → 命令面板含 “Acknowledge All”、“Clear”；
> - “趋势图”视图 → 命令面板含 “Zoom In”、“Export Data”；
> - “只读状态页” → 命令面板为空。

#### 2.2 局部命令不得放入命令面板
- **作用范围有限**的操作（如编辑某个字段、展开某一行详情）**必须放在信息面板内部**；
- 这样可清晰区分：
  - **命令面板**：全局/通用操作（影响整个视图或系统）；
  - **信息面板内控件**：局部/上下文操作（仅影响特定元素）。

> 🚫 禁止：在命令面板放置“编辑第 3 行参数”这类局部操作。

#### 2.3 禁止放置导航按钮
- **任何用于切换视图或功能区域的按钮**（如“Go to Setup”）**不得出现在命令面板**；
- 导航功能**仅由导航面板（底部）或子导航（标签页）负责**。

> ⚠️ 原因：防止用户混淆“执行命令”与“切换界面”。

---

### 3. 复用与布局建议
#### 3.1 视图间可复用命令面板
- 若多个视图**共享完全相同的命令集**，可**共用同一个命令面板定义**；
- 但前提是这些命令**确实适用于所有相关视图**。

> ✅ 合理复用可减少开发维护成本，但不得牺牲上下文准确性。

#### 3.2 避免多列按钮（推荐）
- **强烈建议命令面板仅使用单列按钮**；
- 多列会：
  - 压缩信息面板宽度；
  - 导致不同视图的信息区域**宽高比不一致**；
  - 引起内容“跳动”，分散用户注意力。

> 💡 若命令过多，应考虑：
> - 将部分命令移入信息面板（作为上下文操作）；
> - 或通过“更多…”按钮展开次要命令（需谨慎评估可用性）。

---

### 4. 总结：命令面板设计原则

| 原则           | 说明              |
| ------------ | --------------- |
| **位置可切换**    | 右侧（默认）或左侧（左手模式） |
| **内容上下文绑定**  | 每个视图有自己的命令集     |
| **无命令则为空**   | 不强制填充占位符        |
| **禁止导航功能**   | 视图切换由导航面板处理     |
| **局部操作放信息区** | 保持命令面板“全局性”     |
| **单列优先**     | 避免多列影响布局一致性     |

## 总结
### 1. 核心布局
- **四面板结构**：界面必须包含四个固定区域：
    1. **标题面板**：**顶部**，始终显示。
    2. **信息面板**：**中央**，显示主要内容。
    3. **导航面板**：**底部**，始终显示，用于功能区切换。
    4. **命令面板**：**右侧**（默认）或 **左侧**（左手模式），用于当前视图的操作。
- **安全限制**：主显示界面**禁止关闭、最小化或移动窗口**，防止操作员误触导致界面消失。
- **触摸优化**：所有设计均需**为触摸屏优化**，无需键鼠。

### 2. 导航模型
- **扁平化**：所有功能区域（如 Jobs, Alarms）**处于同一层级**，可任意横向切换。
- **单面板显示**：任何时候**只显示一个信息面板**。
- **详情用对话框**：查看详细信息**必须使用对话框**，**不得新增视图**或页面。
- **子导航**：若功能区需多视图，**所有功能区必须统一**使用 **Tabs (标签页)** 或 **专用按钮区**，且子导航必须与命令面板分离。

### 3. 标题面板 (顶部)
- **始终显示**：位于顶部，**始终可见**。
- **强制内容与布局**：
    - **左上角**：**日期/时间** (若有主机通信，则通信状态在最左侧)。
    - **日期右侧**：**当前视图名称**。
    - **上述内容下方**：**消息显示区**。
    - **右上角**：**登录/登出** 按钮 (若启用安全功能)。

### 4. 导航面板 (底部)
- **始终可用**：位于底部，**始终可见**，**任何对话框（除登录外）不得遮挡**。
- **安全优先**：用户**必须能随时点击导航按钮（尤其是 Alarms）**，无需关闭当前对话框。
- **按钮要求**：**必须包含文本标签**，图标可选。
- **固定位置**：
    - **Alarms**：**倒数第二个** (按钮周围间距应更大)。
    - **Help**：**最右侧**。
- **状态高亮**：
    - **中蓝色**：**当前视图** 或 其他视图有 **未完成任务**。
    - **鲜红色**：**新报警** (Alarms 按钮)。
    - **鲜黄色**：**新警告** (Alarms 按钮，红色优先)。
    - **中绿色**：**就绪状态** (Jobs 按钮)。

### 5. 命令面板 (右侧/左侧)
- **内容绑定**：**仅包含当前信息面板视图相关的通用命令**。
- **严格禁止**：
    1. **禁止放置导航按钮** (如“去设置”)。
    2. **禁止放置局部操作** (如“编辑某行”)，局部操作应在信息面板内。
- **布局**：**强烈建议使用单列按钮**，以保持信息面板宽度一致。

### 6. 对话框 (临时窗口)
- **交互**：**不可移动、不可缩放**。
- **模态规则**：弹出时**不得遮挡标题面板和导航面板**。触发源 (信息/命令面板) **必须禁用**。
- **结构**：必须包含 **标题栏**、**内容区** 和 **底部按钮区**。
- **核心按钮语义**：
    - `OK`: 接受输入或选择 (必填项未完时**禁用**)。
    - `Cancel`: 撤销操作，**恢复原状态**并关闭。
    - `Close`: **仅用于信息展示**，或操作**已执行且无法恢复原状态**时 (严禁此时用 `OK`)。
    - `Yes / No`: **必须成对**用于回答问题。
    - `Apply`: 应用设置，**但不关闭对话框**。
- **类型与按钮**：
    - **信息对话框** (只读)：必须用 **`Close`** (禁止用 `OK`)。
    - **数据输入/选择**：必须用 **`OK` / `Cancel`**。
    - **消息对话框** (带图标)：根据内容使用 `Close`、`OK` 或 `Yes/No`。

### 7. 按钮与文本
- **触摸尺寸**：最短边**不应小于 1.5 厘米**。
- **文本规范**：按钮标签**必须使用“首字母大写” (Title Case)**，例如 `Start Process`。
- **禁止**：**禁止使用全大写** (如 `START`)。
- **突出显示**：
    - **鲜红色**：**报警 (Alarm)**
    - **鲜黄色**：**警告 (Warning)**
    - **中蓝色**：**正在处理 / 未完成**
    - **中绿色**：**需要或请求用户注意**
